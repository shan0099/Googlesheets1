COMMANDS IN GITHUB
81
git intinitializes a brand new Git repository and begins tracking an existing directory. It adds a hidden subfolder within the existing directory that houses the internal data structure required for version control.
82
git clone creates a local copy of a project that already exists remotely. The clone includes all the project's files, history, and branches.
83
git add stages a change. Git tracks changes to a developer's codebase, but it's necessary to stage and take a snapshot of the changes to include them in the project's history. This command performs staging, the first part of that two-step process
84
git commitsaves the snapshot to the project history and completes the change-tracking process. In short, a commit functions like taking a photo. Anything that's been staged with git add will become a part of the snapshot with git commit.
85
git status shows the status of changes as untracked, modified, or staged.
86
git branchshows the branches being worked on locally
87
git merge merges lines of development together. This command is typically used to combine changes made on two distinct branches. For example, a developer would merge when they want to combine changes from a feature branch into the main branch for deployment.
88
git pullupdates the local line of development with updates from its remote counterpart. Developers use this command if a teammate has made commits to a branch on a remote, and they would like to reflect those changes in their local environment
89
git push updates the remote repository with any commits made locally to a branch.

When the work is complete, a branch can be merged with the main project. You can even switch between branches and work on different projects without them interfering with each other.

CONFLICTSA conflict in GitHub is a situation where Git cannot merge two versions of the same file automatically. This can happen when multiple people are working on the same project and make different changes to the same file. When a conflict occurs, Git marks the file as Unmerged and shows both versions of the file in the code. The user needs to manually edit the file and choose which version to keep or combin
113
114
1. Pull the latest changes: Before attempting to merge your changes, make sure you have the latest
version of the repository by pulling the changes from the remote repository
115
 * git pull origin branch-name *
116
2. Identify the conflicts: After pulling the latest changes, Git will notify you of any conflicts. Conflicting
lines of code will be marked with special markers such as "<<<<<<< HEAD" and ">>>>>>> branch-name"
. These markers indicate the conflicting sections that need to be resolved.
117
3. Resolve the conflicts: Open the conflicting file(s) in a text editor and locate the conflicting sections
marked by the markers. Edit the code to resolve the conflicts manually, keeping the changes that are
desired. Remove the conflict markers and ensure the code is in the desired state.
118
4. Stage the resolved changes: After manually resolving the conflicts, stage the modified files to prepare them for commit.
119
* git add file1 file2 ...*
120
5. Commit the changes: Create a new commit with a message explaining the conflict resolution.
121
* git commit -m "Resolve conflicts" *
122
6. Push the changes: Finally, push the changes to the remote repository to make them available to other contributors.
123
* git push origin branch-name *
124
It's important to note that conflicts are a natural part of collaborative development, and effective
communication among team members is crucial to resolving conflicts in a timely manner. Additionally,
it's a good practice to pull the latest changes frequently to minimize conflicts and ensure smoother
collaboration.

APIAPIs allow developers to sync data between multiple platforms and can facilitate communication among the various microservices in web applications. API integration is what does the work when, for example, you enter a new contact in Salesforce and it auto-populates to Marketo as well.
157
TYPES OF APIOpen APIs
Public APIs (or Open APIs) function through a set of tools, routines, and protocols for developing software applications. They allow for access to application data while shielding developers from having access to shared source codes.
158
Partner APIs
Partner APIs are specialized interfaces that enable organizations to access data and service offerings across businesses (B2B). This streamlined approach empowers businesses to create unique features within their own applications or services by utilizing a partner’s resources.
159
Internal APIs
As the name implies, an internal API is a type of API intended for use internally by the organization’s own developers. These APIs facilitate the transmission of data between different components of a system, enabling process automation.
160
Composite APIs
A composite API amalgamates multiple APIs into one interface and provides developers with a unified view of the data from different sources. This integration simplifies data access and utilization by easing interactions with complex applications, commonly seen in connecting various cloud services.
